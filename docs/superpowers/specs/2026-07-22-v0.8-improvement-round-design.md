# v0.8 Improvement Round — Design

**Date:** 2026-07-22
**Status:** Approved in brainstorming; awaiting spec review
**Goal:** Make the plugin genuinely language-agnostic, turn the context meter into a project
dashboard that works on any machine, make pipeline-architect's generated pipelines "CI/CD
proper", and move two methodology promises from prose into harness-enforced hooks.

Per the project owner's working preference, every notable decision below carries a
**cost** (effort/complexity/maintenance) and **effectiveness** (how well it solves the
problem) assessment, and higher-cost options were chosen where effectiveness justified it.

---

## 1. Context-meter v3

### 1.1 Dual runtime (portability)
- New `skills/context-meter/scripts/context_meter.mjs`: zero-dependency Node ≥ 18 port
  of the meter, feature-identical to the Python version.
- The existing `context_meter.py` stays and gains the same v3 features.
- **Parity is a tested invariant:** a golden-fixture suite — `tests/fixtures/context-meter/`
  holding `<case>.input.json` → `<case>.expected.txt` pairs — runs against **both**
  implementations in CI. Drift between the two is a red build, not a doc footnote.
- *Cost: med-high (two codebases). Effectiveness: high (meter works wherever either
  runtime exists; fixtures make the duplication safe). Chosen over Node-only (breaks
  users without Node) and Python-only (doesn't remove the Python requirement).*

### 1.2 Agent-led setup
- SKILL.md's "Enable it" section becomes an install flow the agent performs:
  1. Probe runtimes in order: `node` → `python` → `python3` → `py`.
  2. Read the target `settings.json` (user or project — ask which); if a `statusLine`
     block exists, never clobber — ask before replacing or chaining.
  3. Write the `statusLine` block pointing at the detected runtime + script.
  4. **Verify:** pipe a bundled sample JSON through the exact configured command and
     confirm a traffic-light line comes back before declaring success.
- The "Windows: use python / macOS: python3" user-facing note is removed — detection
  replaces it.
- *Cost: low. Effectiveness: high (setup goes from hand-editing JSON to one approval).*

### 1.3 Richer display
Target line (all segments present):

```
🟡 71% (142k/200k) · red ~35m · opus · main · step 3/7 · 23 turns · 1h42m · $0.42 · $0.25/h ↺1
```

- **Model** — short model name from the stdin JSON's model block (already in the payload).
- **Git branch** — read `.git/HEAD` directly from the workspace directory (no subprocess).
  Handles worktrees (`gitdir:` file indirection) and detached HEAD (short SHA).
- **Step `3/7`** — when `workflow.json` exists in the project root: first `in_progress`
  step over total (fallback: first non-`done`). Cached by file mtime in the state file.
  Absent file → segment absent; the meter stays universal.
- **Cost + burn rate** — `$0.42 · $0.25/h`; burn rate appears once session age ≥ 10 min
  (earlier is noise). Cost segment still feature-detected (older Claude Code sends none).
- **Opt-out** — `CLAUDE_CONTEXT_METER_HIDE=branch,model,…` env var hides named segments
  for narrow panes. No other configuration surface.
- *Cost: med. Effectiveness: high — the meter becomes a step-wise project dashboard.*

### 1.4 Smarter predictions
- **Idle-aware burn rate:** inter-sample gaps > 10 min are excluded from the time span
  used for `red ~Xm`, so breaks stop inflating the estimate. *(Cost: low; Effectiveness:
  high — the largest real-world error source today.)*
- **Recency-weighted growth:** the next-turn estimate uses a recency-weighted median so
  a shift into heavy tool use shows quickly, while one huge turn still can't dominate.
  *(Cost: low; Effectiveness: med.)*
- **Learned red line:** when a compaction is observed, record the usage % at which it
  actually happened; the effective red threshold becomes
  `min(YELLOW_BELOW, observed_compaction_pct − safety_margin)` for that machine's state.
  Constants stay overridable at the top of both scripts. *(Cost: med; Effectiveness:
  med-high — replaces a guessed constant with observed truth.)*

## 2. De-Python-izing the plugin surface

- **Docs renderer dual runtime:** `render_docs.mjs` alongside `render_docs.py`, same
  golden-fixture parity pattern (decisions/workflow JSON in → HTML out), and
  `browser-readable-project-docs` SKILL.md gains the same runtime auto-detection.
  *(Cost: med; Effectiveness: med-high.)*
- **Gate command matrix:** `pipeline-architect` gains a reference table mapping each gate
  (unit tests, lint, format, type check, coverage + floor, install/build) to vetted
  commands per ecosystem: Python (pytest/ruff/mypy), Node+TS (npm test/eslint/prettier/
  tsc), Go (go test/golangci-lint/gofmt), Rust (cargo test/clippy/fmt), generic/other
  (project-provided commands). The Stage-B subagent tailors `ci.yml` from this matrix
  instead of improvising. The template's "steps assume Python" comment is replaced by a
  pointer to the matrix. *(Cost: med. Effectiveness: high — chosen over per-language
  template files: higher upkeep surface, no added effectiveness since the subagent
  composes final YAML anyway.)*
- **Phrasing sweep:** audit all SKILL.md files for Python-flavored assumptions
  (interpreter notes, `pip install`, pytest-specific caveats presented as universal) and
  neutralize or generalize. *(Cost: low; Effectiveness: med.)*

## 3. Pipeline-architect: generated pipelines become CI/CD proper

Unit tests already block at **L0** in every generated pipeline (with UAT/SIT at L1 and
the coverage ladder above); this section closes the remaining gaps:

- **Per-archetype delivery templates.** Today only release-on-tag ships as YAML; the
  other archetypes' delivery halves are prose the subagent must improvise. Ship vetted
  templates: `delivery-gated-deploy.yml` (build → staging → prod behind a protected
  GitHub Environment), `delivery-preview-prod.yml` (per-PR preview + gated prod),
  `delivery-publish.yml` (artifact + metrics to a GitHub Release on the human's tag).
  Production remains behind manual approval in every one — the approval gate is
  non-negotiable. *(Cost: med-high; Effectiveness: high — the difference between "CI
  with a deploy paragraph" and real CD.)*
- **CI hygiene defaults** in the skeleton: `concurrency` group with
  `cancel-in-progress` for PRs, dependency caching, `timeout-minutes` on jobs, actions
  pinned to major versions. *(Cost: low; Effectiveness: med-high.)*
- **Test-matrix rung:** for library/CLI/plugin archetypes, the rubric offers an OS ×
  language-version matrix at L2+. Not a default for services. *(Cost: low-med;
  Effectiveness: med.)*
- **Dogfooding (this repo):** branch protection on `main` requiring a green `gates`
  check (pipeline-architect Stage D); a `release` job creating a GitHub Release from the
  matching `CHANGELOG.md` section when the human pushes `vX.Y.Z`; the repo's own
  `pipeline.md` recording archetype (release-on-tag) and level; CI extended to run the
  golden fixtures under **both** Node and Python — the repo tests its own portability
  claim. First tag: `v0.8.0` when this round ships. *(Cost: low-med; Effectiveness:
  high.)*

## 4. Expert-review adoptions (all three packs approved)

From the Anthropic-plugin-expert review (which also verified `plugin.json` is well-formed
and `context_meter.py`'s stdin field names are correct against current Claude Code docs):

### 4.1 Hooks pack — prose promises become harness guarantees
- **PreToolUse secret-scan gate:** plugin hook that scans staged changes before
  commit-capable tool calls, so a secret cannot be committed even if the model forgets
  the skill. `secret-safe-commits` SKILL.md remains as the why + manual procedure.
  *(Effectiveness: high; Cost: med.)*
- **SessionStart state restore:** hook detects `plan.md` / `workflow.json` /
  `pipeline.md` / `HANDOFF.md` and injects a brief state summary at session start,
  making stepwise re-entry reliable instead of remembered. *(Effectiveness: med-high;
  Cost: med.)*
- **Context-meter setup nudge:** the same SessionStart hook notes when no `statusLine`
  is configured and points at the context-meter skill (detection trigger for §1.2's
  install flow — not an auto-install). *(Effectiveness: low-med; Cost: low.)*

### 4.2 Structure pack
- **Ship pipeline-architect as a real plugin agent** (`agents/pipeline-architect.md`)
  instead of a copy-me template — discoverable, versioned with the plugin, declared
  tools/permissions. SKILL.md references the agent by name. *(Effectiveness: med;
  Cost: low.)*
- **Two-gate doctrine** documented in `stepwise-delivery`: the **commit gate**
  (deterministic, hook-enforceable) vs the **advance gate** (semantic, human-only,
  never automated). Prevents future auto-advance "improvements" from breaking the
  methodology's core invariant. *(Effectiveness: med; Cost: low.)*

### 4.3 Polish pack
- `plugin.json`: add `displayName: "Step-Wise Methodology"`; keep the plugin enabled by
  default (current behavior — the expert floated `defaultEnabled: false`, but the skills
  only activate on matching tasks, so auto-enable costs users nothing).
  *(Effectiveness: low; Cost: low.)*
- **Sharper skill triggers:** frontmatter descriptions gain not-for clauses and alias
  phrasings where matching is currently situational (e.g. `craftsmanship-bar`).
  *(Effectiveness: low-med; Cost: low.)*
- **README quick-start flowchart** showing which skill applies when, instead of any
  skill restructuring. *(Effectiveness: low; Cost: low.)*

**Explicitly rejected:** merging/consolidating skills by phase (high disruption, low
ROI — expert and owner agree); shipping the status line as a plugin.json component
(not supported by the schema — verified).

## Testing strategy

- Golden fixtures for both dual-runtime scripts, run against both implementations in CI.
- Existing structural test suite (`tests/`) extends to cover: new templates parse as
  valid YAML, hooks config well-formed, agent frontmatter valid, gate matrix table
  complete per ecosystem.
- Meter unit behaviors (idle-gap exclusion, learned red line, branch/step parsing) get
  fixture cases, including worktree `.git` files and detached HEAD.
- First green run on CI (both runtimes) gates the `v0.8.0` tag.

## Out of scope (this round)

- Any auto-advance of steps or auto-deploy to production (violates the advance gate).
- Skill consolidation/restructuring.
- Compiled-binary distribution of the meter.
- New methodology skills.

## Delivery order (suggested for the implementation plan)

1. §4.3 polish + §4.2 structure (small, unblock nothing)
2. §1 context-meter v3 (fixtures first, then Node port, then features in both)
3. §2 de-Python (renderer port, gate matrix, sweep)
4. §3 pipeline-architect upgrades (templates, hygiene, matrix rung)
5. §4.1 hooks pack
6. Dogfooding: CI upgrade → branch protection → `pipeline.md` → tag `v0.8.0`
